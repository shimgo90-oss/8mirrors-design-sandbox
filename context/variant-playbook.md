# 변형 플레이북 — 랜딩 A/B 만드는 법

> 마케터(Codex) 레인의 핵심 가이드. "어떻게 만드나"는 [app/landing/_variants.tsx](../app/landing/_variants.tsx) 상단 주석 참고. 이 문서는 "무엇을·왜" 테스트하나.

## 좋은 A/B 테스트의 원칙
1. **한 번에 하나의 가설.** 헤드라인 *그리고* 순서 *그리고* CTA를 동시에 바꾸면 뭐가 이겼는지 모름. 변형 간 차이를 좁게.
2. **가설을 `note`에 적어라.** "왜 이게 더 잘 될 거라 보는가". 결과 해석의 기준.
3. **같은 광고 → 다른 슬러그.** 트래픽 소스를 통제해야 비교가 유효.
4. **베이스라인(original) 유지.** 항상 기준점과 비교.

## 무엇을 테스트할 만한가 (축)
- **메시지/포지셔닝**: 루틴 중심 vs 결과/변화 중심 vs 전문가 검토 중심 → `copy` 오버라이드로.
- **길이/구조**: 풀버전 vs 린(빠르게 CTA) → `sections` 순서·부분집합으로.
- **고민 진입점**: redness 중심 vs 일반 → 커스텀 레이아웃(Claude 작업).
- **CTA 문구**: "Build my routine" vs "Try it free" vs "See what fits my skin".

## 카피 규칙 (반드시)
- 영어(US 30대 여성). [[brand-voice]]·[[audience]] 준수.
- **'리포트'로 팔지 말 것.** 맞춤 루틴·결과로. [[value-prop]]
- 라임 하이라이트는 섹션당 1곳, 검정 텍스트. 시안 CTA는 검정 텍스트. (DS: ../DESIGN.md)
- 광고 클리셰·느낌표 남발 금지.

## 레인 경계 (충돌 방지)
- **마케터(Codex)가 하는 것**: `_variants.tsx`에서 변형 추가/복제, `copy`로 문구 변경, `sections` 순서 조정. = config·content만.
- **Claude(개발)에 넘기는 것**: 새로운 섹션·블록, 커스텀 레이아웃(`Custom`), 컴포넌트 수정, 렌더 버그. = React 코드.
- 규칙: **마케터는 React 컴포넌트를 새로 만들지 않는다.** 필요하면 GitHub 이슈/요청으로 넘긴다.

## 배포·검토 흐름
1. `variant/<slug>` 브랜치에서 작업 → push → Vercel 프리뷰 URL 자동 생성.
2. 프리뷰에서 직접 확인(모바일 실측 권장).
3. PR 올림 → **고고가 디자인 관점 검토 → 머지**(마케터는 main 직접 머지 X).
4. 머지 = production 배포 → 광고에 라이브 URL 연결.
